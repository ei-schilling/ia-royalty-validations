# Royalty Statement Validator — Agent Configuration

## Project Overview

This is a **Royalty Statement Validator** — a standalone web application that validates
royalty statement files against Schilling ERP business rules. It consists of a Python/FastAPI
backend, a React/TypeScript frontend, and a PostgreSQL database, all orchestrated with Docker.

## Architecture

```
royalties/
├── backend/          # Python 3.12+ / FastAPI REST API
│   ├── app/
│   │   ├── api/          # Route handlers (auth, uploads, validations)
│   │   ├── models/       # SQLAlchemy ORM models
│   │   ├── schemas/      # Pydantic request/response schemas
│   │   ├── services/     # Business logic layer
│   │   ├── validation/   # Validation engine + rule plugins
│   │   │   ├── rules/    # One file per validation rule (11 rules)
│   │   │   ├── engine.py # Rule orchestrator
│   │   │   ├── parser.py # Multi-format file parser (PDF, CSV, Excel, JSON)
│   │   │   └── base_rule.py
│   │   ├── db/           # Database setup (async SQLAlchemy + PostgreSQL)
│   │   └── config.py     # Environment-driven settings
│   └── tests/            # pytest test suite
├── frontend/         # React 19 + TypeScript + Vite + Tailwind + shadcn/ui
│   ├── src/
│   │   ├── pages/        # LoginPage, UploadPage, ResultsPage
│   │   ├── components/   # Reusable UI components
│   │   ├── api/          # API client wrappers
│   │   ├── hooks/        # Custom React hooks
│   │   └── types/        # TypeScript interfaces
│   └── tests/            # Vitest unit tests
└── docker-compose.yml
```

## Tech Stack

- **Backend**: Python 3.12, FastAPI, SQLAlchemy 2.0 (async), PostgreSQL 16, pdfplumber, openpyxl, pandas
- **Frontend**: React 19, TypeScript, Vite, Tailwind CSS, shadcn/ui
- **Testing**: pytest (backend), Vitest (frontend), Playwright (E2E)
- **Infrastructure**: Docker, docker-compose

## Code Standards

- All code documentation in **English**
- Use `Annotated` style for FastAPI dependencies (official FastAPI convention)
- Use `async def` only when truly async; use plain `def` for blocking/sync operations
- Pydantic models use `model_config = {"from_attributes": True}` for ORM compatibility
- Validation rules extend `BaseRule` abstract class with `rule_id`, `description`, `validate()`
- Danish field names preserved in data layer (BELOEB, ANTAL, STKPRIS) to match Schilling exports
- File parsers normalize all formats into `list[dict]` with consistent key names

## Key Business Domain

- **Schilling ERP**: Danish publishing industry ERP system (C++/Oracle)
- **Royalty settlements**: Payments to authors based on book sales
- **PDF format**: "Royalty afregning" PDFs generated by Apache FOP; one page per agreement
- **Danish number format**: Period = thousands separator, Comma = decimal (e.g., 70.470,00 = 70470.00)
- **Transaction types**: 40 types defined in Roy.h (Salg, Retur, GarGlobal, Forskud, etc.)

## Testing Requirements

- Every phase must have tests that pass 100% before moving to the next phase
- Use pytest fixtures in `conftest.py` for shared test state
- Test files use in-memory SQLite for database tests
- Use `httpx.AsyncClient` for API endpoint tests
- Fixture files in `tests/fixtures/` for sample royalty statements
