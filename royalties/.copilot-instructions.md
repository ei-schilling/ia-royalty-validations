# Copilot Instructions — Royalty Statement Validator

## Language & Documentation

- All code comments, docstrings, variable names, commit messages, and documentation must be in **English**
- Danish field names from the Schilling ERP domain are preserved as-is (e.g., `BELOEB`, `ANTAL`, `STKPRIS`, `AFTALE`)
- Use clear, concise docstrings for all public functions and classes

## Backend Conventions (Python / FastAPI)

### FastAPI

- Use `Annotated` for all parameter declarations and dependencies
- Create type aliases for reusable dependencies: `DbSession = Annotated[AsyncSession, Depends(get_db)]`
- Router-level prefix and tags: `router = APIRouter(prefix="/api/auth", tags=["auth"])`
- Return types on all endpoints; use `response_model` only when filtering
- Sync `def` for blocking operations; `async def` only with truly async logic
- One HTTP operation per function

### Pydantic

- Use `model_config = {"from_attributes": True}` for ORM-backed schemas
- No `...` (Ellipsis) for required fields
- Use `Field()` with constraints (min_length, ge, le, etc.)

### SQLAlchemy

- Async sessions via `async_sessionmaker` with `expire_on_commit=False`
- `Mapped` type annotations for all columns
- UUID primary keys with `default=uuid.uuid4`
- Timezone-aware datetimes with `DateTime(timezone=True)`

### Validation Rules

- Each rule is a separate file in `app/validation/rules/`
- All rules inherit from `BaseRule` with `rule_id`, `description`, `validate()` methods
- Return `list[ValidationIssue]` — empty means rule passed
- Use `Severity.ERROR`, `Severity.WARNING`, `Severity.INFO`
- Include `context` dict with agreement/product info for traceability

### File Parsing

- All parsers normalize output to `list[dict]` with consistent key names
- Every row dict includes `_row_number`, `_source`, and optionally `_record_type`, `_page_number`
- Danish number conversion: `_danish_number_to_str()` converts "70.470,00" → "70470.00"
- Schilling CSV formulas: `=N/100` resolved by `_resolve_formula()`

## Frontend Conventions (React / TypeScript)

### React

- Functional components only
- Custom hooks for data fetching (`useUpload`, `useValidation`)
- TypeScript strict mode
- shadcn/ui components for consistent design
- Tailwind CSS for styling — no custom CSS files unless absolutely necessary

### State Management

- React state + context for simple state
- No Redux — the app is small enough for built-in React patterns

### API Client

- Fetch-based API client in `src/api/client.ts`
- All API calls return typed responses
- Error handling at the hook level

## Testing Standards

- **Backend**: pytest with `asyncio_mode = "auto"`, fixtures in `conftest.py`
- **Frontend**: Vitest for unit tests, Playwright for E2E
- **Coverage**: Aim for 90%+ on validation rules
- **Fixtures**: Sample files in `tests/fixtures/` (CSV, Excel, JSON, PDF)
- Tests must pass 100% before moving to the next implementation phase

## Docker

- Multi-stage Dockerfiles for both backend and frontend
- Non-root user in production images
- `docker-compose.yml` with services: db (PostgreSQL), backend (FastAPI), frontend (React)
- Health checks on all services
